1.1 背景与目的

为推进AI原生研发范式在全公司的落地，PMO组织对15个研发团队/部门进行项目上下文核查。本次核查覆盖三大目标、九项核心指标，旨在评估各团队在统一架构与规范、团队共识与遵从、AI可用性闭环方面的成熟度，识别短板并提供建议。

1.2 核查范围与方法

核查范围：15个研发团队/部门（详见部门编号对照表）。

核查方式：

3.3 共性发现

📐 目标1：统一架构与规范

  • Inform L2 已普遍达成：Checklist 推动了入口、导航与目录分层，多数团队 AI 不再需全仓盲搜。
  • 六维共性短板：Verify（断链/占位词 CI）和 Correct（last_verified_at + 过期降级）几乎全员缺失；Observe（断链率、stale 率、阻塞原因）仅个别团队萌芽。
  • 结构标杆 ≠ 维护成熟：多团队目录结构完整（Inform L2），但无保鲜机制、无 CI 门禁，长期维护靠人肉（Entropy L1）。
  • 单文档多维度复用是可行路径：编号9（BrowserAct）证明不必为每层独立建文件，但需配套抗熵机制。

🤝 目标2：团队共识与遵从

  • 团队知晓度两极分化：部分团队通过群公告、onboarding 文档有效宣导，但仍有5个部门团队知晓为"无"。
  • 文档更新 SLA 缺失：多数团队无明确的文档更新频率要求，导致"近期入库"项证据不足。
  • 多人贡献机制薄弱：部分团队仅1-2人维护文档，缺乏 Code Review 或多人协作机制。
  • 举证质量参差：部分团队举证图片与核查项无关（如运维通知、API文档），需提升举证规范性。

🤖 目标3：AI可用性闭环

  • AI工具接入率较高：多数团队已配置 .cursor/rules/ 或 AGENTS.md，但部分团队缺配置文件截图证据。
  • AI实际消费案例不足：部分团队自评"有"但举证图片为静态文档，非 AI 实际引用案例。
  • 上下文层级设计缺失：多数团队未展示 AI 上下文层级设计（如不同层级引用不同文档）。
